 当前系统管理员认证存在两个安全缺口：
     1. 无登录失败限制 — 攻击者可无限次暴力破解管理员密码
     2. 无会话互斥 — 管理员可从多设备同时登录，无法检测旧会话劫持

     此次变更引入两项安全加固：密码重试锁定（3次输错锁定30分钟）和单会话强制（新登录踢旧会话），均通过数据库记录实现服务端状态追踪。

     实施详情

     1. config.py — 新增配置常量

     在 Config 类中，ADMIN_TOKEN_EXPIRY_MINUTES = 10 行后添加：

     # 管理员登录安全
     MAX_LOGIN_ATTEMPTS = 3          # 连续失败 N 次后锁定
     LOGIN_LOCKOUT_MINUTES = 30      # 锁定时长（分钟）

     2. models/admin.py — Admin 模型扩展

     新增列（created_at 行后）：
     failed_attempts = db.Column(db.Integer, default=0)
     locked_until = db.Column(db.DateTime)
     session_token = db.Column(db.String(64))

     新增方法 is_locked() — 检查是否锁定中，若锁定已过期则自动清除；返回 bool。

     新增方法 lockout_remaining_minutes() — 返回剩余锁定分钟数（向上取整，最小 1）。

     3. routes/auth.py — admin_login 重写（核心）

     在现有函数体内插入以下逻辑：

     1. admin not found 后 → 调用 admin.is_locked()，若锁定返回 HTTP 423 + 剩余分钟数
     2. 密码验证失败分支 → failed_attempts += 1，若达到阈值则设置 locked_until 并返回 423；否则返回 401 + remaining_attempts
     3. 密码验证成功分支 → 重置 failed_attempts=0, locked_until=None，生成 session_token = uuid4() 写入 DB，放入 JWT payload

     JWT payload 新增字段：'session_token': admin.session_token

     4. routes/auth.py — admin_required 装饰器加固

     在 role != 'admin' 检查通过后、pw_expired 检查前插入：

     jwt_session_token = g.current_user.get('session_token')
     if jwt_session_token:
         admin = Admin.query.get(g.current_user.get('user_id'))
         if admin and admin.session_token and admin.session_token != jwt_session_token:
             return jsonify({
                 'code': 409,
                 'message': '管理员已在其他设备登录，您已被强制下线。',
                 'session_conflict': True,
             }), 409

     使用 HTTP 409 区分"会话被踢"与 401"令牌过期"。session_conflict: True 供前端精确检测。

     5. routes/admin.py — change_password 携带 session_token

     在 generate_jwt 调用的 payload 中新增：'session_token': admin.session_token

     确保密码修改不会导致会话被误判为失效。

     6. migrations/003_admin_security.sql — 新建迁移脚本

     ALTER TABLE admins
       ADD COLUMN failed_attempts INT DEFAULT 0,
       ADD COLUMN locked_until DATETIME NULL,
       ADD COLUMN session_token VARCHAR(64) NULL;

     7. migrations/init.sql — 更新全新安装表结构

     admins 表定义中增加上述三列（含默认值）。

     8. admin/login.html — 前端错误消息细化

     submit 事件处理中，非成功分支新增：
     - data.code === 423 → 显示锁定消息
     - data.remaining_attempts !== undefined → 显示剩余尝试次数
     - 其他 → 保持原有逻辑

     9. admin/index.html — api() 检测会话冲突

     在现有 r.status === 401 判断之前新增：
     if (r.status === 409) {
       return r.json().then(body => {
         if (body.session_conflict) {
           alert(body.message);
           localStorage.clear();
           window.location.href = '/admin/login.html';
         }
         return { code: 409, message: body.message, session_conflict: true };
       });
     }

     验证方法

     功能验证

     1. 锁定流程：连续 3 次输错密码 → 验证返回 HTTP 423 + 锁定 30 分钟 → 用正确密码尝试 → 仍被拒绝
     2. 剩余次数：输错 1 次 → 验证返回 remaining_attempts: 2
     3. 锁定过期：手动将 DB locked_until 设为过去时间 → 再次登录 → 锁定自动清除
     4. 成功重置：输错 2 次后成功登录 → 验证 failed_attempts 重置为 0
     5. 会话互斥：浏览器 A 登录 → 浏览器 B 登录同一账号 → 浏览器 A 任意 API 调用返回 409 → 弹窗 "管理员已在其他设备登录，您已被强制下线。" → 跳转登录页
     6. 密码修改不中断：修改密码后 session_token 不变 → 新 JWT 正常工作
     7. Token 过期不误判：JWT 自然过期 → 返回 401（非 409）

     运行验证

     - 启动 Flask 后端：cd backend && python app.py
     - 浏览器打开 http://localhost:5000/admin/login.html
     - 执行上述功能验证步骤
     - 直接检查 DB 字段：SELECT username, failed_attempts, locked_until, session_token FROM admins;

  安全性提升

  功能 1 — 密码重试锁定： 连续 3 次输错 → 账户锁定 30 分钟 → 锁定过期自动清除 → 成功后计数器清零

  功能 2 — 单会话强制： 新设备登录生成新 session_token → 旧设备的 JWT 中 session_token 与 DB 不匹配 → 返回 409 → 前端弹窗并跳转登录页
